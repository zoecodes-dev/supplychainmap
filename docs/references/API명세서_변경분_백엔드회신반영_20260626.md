## ② DPP(§6) → 기존 경로·키(dpp_id, /records, 제품 단위)로 정정

발행 엔드포인트가 두 벌로 갈라지는 것을 막기 위해 신규(🆕)에서 백엔드 기존(🔵)/재합의(🟡)로 변경.

**Before**
```markdown
### 6.1 발행 이력
| 6.1a | GET | /dpp-history?destination=&approved_by=&status=&start_date=&end_date= | 🆕 |
| 6.1b | GET | /dpp/{dppId} | 🆕 |

### 6.3 DPP Readiness · 발행
| 6.3a | GET | /dpp/readiness | 🆕 |
| 6.3b | GET | /dpp/readiness/{serialNumber} | 🆕 |
| 6.3c | POST | /dpp/{serialNumber}/issue | 🆕 |
```

**After**
```markdown
### 6.1 발행 이력
| 6.1a | GET | /dpp/records?destination=&approved_by=&status=&start_date=&end_date= | 🔵 백엔드 기존 |
| 6.1b | GET | /dpp/records/{dppId} | 🟡 재합의(단건 경로 확인) |

### 6.3 DPP Readiness · 발행
| 6.3a | GET | /dpp/products/{productId}/readiness | 🔵 백엔드 기존(제품 단위) |
| 6.3b | GET | /dpp/readiness?status=hold (보완 대기 목록) | 🟡 재합의(목록 경로 미정) |
| 6.3c | POST | /dpp/{dppId}/issue | 🔵 백엔드 기존(UUID 키) |
```

핵심 변경:
- `/dpp-history` → **`/dpp/records`**
- 발행/조회 키 `serialNumber` → **`dppId`(UUID)** 로 통일 (불변 Lock 단일화)
- readiness **제품(product_id) 단위**로 정정 (프론트 화면이 쓰던 serialNumber 단위 아님)
- §6 상단에 정정 경고 박스 추가, 6.2b held-products도 `serialNumber` → `productId`

## ③ BOM(§10.1b) → ✅ 기존이 아니라 🟡 재합의

**Before**
```markdown
### 10.1 제품 / BOM (기존)
| 10.1a | GET | /products | ✅ 기존 |
| 10.1b | GET | /products/{productId}/bom | ✅ 기존 |

- 10.1b: ApiProductBom { bomVersions[], parts[], bomItems[] } (lib/api.ts:519~549 ...)
```

**After**
```markdown
### 10.1 제품 / BOM
| 10.1a | GET | /products | ✅ 기존 (단, §0.2 무인증 → P1 인증 배선 대상) |
| 10.1b | GET | /products/{productId}/bom?as_of=YYYY-MM-DD | 🟡 재합의 |

- 10.1b 🔴 재합의 필요:
  - 백엔드 실제: GET /products/{product_id}/bom?as_of=YYYY-MM-DD, 응답은 BOM 트리 객체
  - 프론트 getProductBom은 ?bom_version_id= + 평면 {bomVersions,parts,bomItems} 기대 (주석 "신규-추가예정")
  - 확정 2건: ① as_of vs bom_version_id  ② 트리 객체 vs 평면 3배열
```

## ④ 목록 total 규약(§0.6) → X-Total-Count 헤더로 확정

**Before**
```markdown
### 0.6 페이지네이션
- 쿼리: ?page={1-base}&size={n} + 도메인 필터(snake_case).
- 응답: 기본은 순수 배열. 총건수가 필요하면 헤더 X-Total-Count 또는
  envelope { items, total, page, size } 중 백엔드와 합의(미정).
```

**After**
```markdown
### 0.6 페이지네이션 ✅ 확정 (2026-06-26)
- 쿼리: ?page={1-base}&size={n} + 도메인 필터(snake_case).
- 응답 본문: 순수 배열 그대로 유지(envelope로 감싸지 않음).
- 총건수: X-Total-Count 응답 헤더로 전달(필터 적용 후 전체 건수).
    GET /suppliers?page=1&size=20
    200 OK
    X-Total-Count: 187
    [ ...20건... ]
- 프론트는 페이지 수를 ceil(X-Total-Count / size)로 계산.
- 프록시 주의: rewrite 통과 필요. CORS 직결 시 Access-Control-Expose-Headers: X-Total-Count.
```

> **결정**: 백엔드가 프론트 선호로 넘긴 건 → **A안(X-Total-Count 헤더)** 채택.
> 사유: 모든 목록을 bare array로 통일 유지(envelope 혼재 방지), 프론트 변경 최소.
